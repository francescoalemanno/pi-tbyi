# Triage

Move local Markdown PRDs and issues through a small state machine of labels.

Default locations:

- PRDs: `docs/prds/`
- Issues: `docs/issues/`

Every triage note written into a file must start with this disclaimer:

```markdown
> *This was generated by AI during triage.*
```

## Reference docs

- [AGENT-BRIEF.md](./AGENT-BRIEF.md) — how to write durable agent briefs.
- [OUT-OF-SCOPE.md](./OUT-OF-SCOPE.md) — how the `.out-of-scope/` knowledge base works.

## Local Markdown conventions

PRDs and issues use frontmatter:

```yaml
---
title: Short title
labels:
  - enhancement
  - needs-triage
status: open
---
```

Category labels:

- `bug` — something is broken.
- `enhancement` — new feature or improvement.

State labels:

- `needs-triage` — maintainer needs to evaluate.
- `needs-info` — waiting on requester for more information.
- `ready-for-agent` — fully specified, ready for an AFK coding agent.
- `ready-for-human` — needs human implementation or judgement.
- `wontfix` — will not be actioned.

Every triaged file should carry exactly one category label and one state label. If state labels conflict, flag it and ask the maintainer before editing.

State transitions: an untriaged file normally goes to `needs-triage` first. From there it moves to `needs-info`, `ready-for-agent`, `ready-for-human`, or `wontfix`. `needs-info` returns to `needs-triage` once the requester replies. The maintainer can override at any time.

## Invocation

The user invokes `/skill:triage` and describes what they want in natural language. Examples:

- “Show me anything that needs my attention.”
- “Let's look at `docs/issues/003-login-errors.md`.”
- “Move this issue to ready-for-agent.”
- “What's ready for agents to pick up?”

## Show what needs attention

Inspect `docs/issues/` and `docs/prds/`, then present three buckets, oldest first when file metadata is available:

1. **Untriaged** — missing a state label.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with new comments or user-provided updates** — needs re-evaluation.

Show counts and a one-line summary per file. Let the maintainer pick.

## Triage a specific file

1. **Gather context.** Read the full file, frontmatter, comments, and prior triage notes. Parse prior notes so you do not re-ask resolved questions. Explore the codebase using glossary vocabulary and relevant ADRs. Read `.out-of-scope/*.md` and surface any prior rejection that resembles this request.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the issue. Wait for direction.

3. **Reproduce for bugs.** Before grilling, attempt reproduction: read the reported steps, trace relevant code, run tests or commands. Report successful repro with code path, failed repro, or insufficient detail. A confirmed repro makes a much stronger agent brief.

4. **Grill if needed.** If the issue needs fleshing out, use `/skill:grill-with-docs` behavior: ask one question at a time, update context docs/ADRs when durable language or decisions crystallize.

5. **Apply the outcome:**
   - `ready-for-agent` — add an `## Agent Brief` section using [AGENT-BRIEF.md](./AGENT-BRIEF.md).
   - `ready-for-human` — add the same structure, but note why it cannot be delegated.
   - `needs-info` — add triage notes using the template below.
   - `wontfix` for bugs — add a polite explanation and set `status: closed`.
   - `wontfix` for enhancements — write/update `.out-of-scope/`, reference it in the file, and set `status: closed`.
   - `needs-triage` — apply the label. Optional note if there is partial progress.

## Quick state override

If the maintainer says “move this to ready-for-agent”, trust them and apply the role directly. Confirm what you are about to change, then edit. If moving to `ready-for-agent` without a grilling session, ask whether they want an agent brief.

## Needs-info template

```markdown
## Triage Notes

> *This was generated by AI during triage.*

**What we've established so far:**

- point 1
- point 2

**What we still need from the requester:**

- specific question 1
- specific question 2
```

Capture everything resolved during grilling under “established so far” so the work is not lost. Questions must be specific and actionable.

## Resuming a previous session

If prior triage notes exist, read them, check whether the requester has answered outstanding questions, and present an updated picture before continuing. Do not re-ask resolved questions.
